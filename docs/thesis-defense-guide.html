1. Tổng Quan & Bài Toán

Hệ thống ThangLong University Web Portal (TLU UMS) là một nền tảng tích hợp phục vụ nhu cầu số hóa quy trình quản lý đào tạo tín chỉ, giải quyết các nhu cầu thiết thực của 3 nhóm đối tượng: Sinh viên, Giảng viên và Quản trị viên (Admin Phòng Đào tạo).

Mục tiêu hệ thống

  • Student Portal: Sinh viên có thể quản lý hồ sơ, đăng ký học phần trực tuyến, theo dõi thời khóa biểu, lịch thi, tra cứu điểm số chi tiết (thang điểm 10, thang điểm chữ và hệ 4), thanh toán học phí online thông qua cổng VNPay và chat trực tuyến với giảng viên/lớp học.
  • Teacher Portal: Giảng viên theo dõi lịch dạy học, thực hiện điểm danh điện tử từng buổi học (1-15 buổi), nhập điểm thành phần trực tiếp trên giao diện và thực hiện khóa điểm gửi phòng đào tạo.
  • Admin Portal: Quản trị viên thiết lập cấu trúc học thuật (Major, Department), quản lý chương trình học (Courses, môn tiên quyết), mở học kỳ mới, quản lý lớp học phần, chia phòng thi, quản lý Knowledge Base và kiểm soát lịch sử hệ thống qua Audit Logs.
⚠️ Bài Toán Cốt Lõi: Tải Đỉnh Đột Biến (Peak Traffic)
Trong đào tạo tín chỉ, khi cổng đăng ký môn học mở ra, hàng nghìn sinh viên truy cập đồng thời để giành giật các lớp có lịch học hoặc giảng viên ưa thích. Điều này tạo ra hàng nghìn request/giây ghi trực tiếp xuống CSDL, gây cạn kiệt Connection Pool và sập hệ thống. TLU Portal giải quyết triệt để bài toán này bằng hàng đợi Apache Kafka.

2. Kiến Trúc & Công Nghệ

Hệ thống được phát triển theo mô hình Client-Server, trong đó Backend cung cấp RESTful APIs và WebSocket endpoints, còn Frontend là ứng dụng Single Page App (SPA) tương tác bất đồng bộ.

Backend (Spring Boot 3.x / Java 17)

  • Spring Data JPA (Hibernate): Quản lý ORM và tương tác cơ sở dữ liệu PostgreSQL.
  • Spring Security + JWT: Xác thực và phân quyền phi trạng thái (Stateless).
  • Redis: Caching thông tin phiên làm việc, danh sách user online và lưu trữ Refresh Token.
  • Apache Kafka: Message broker trung gian điều phối hàng đợi đăng ký bất đồng bộ.
  • WebSocket (STOMP): Giao thức truyền tin realtime cho hệ thống chat.

Frontend (TanStack Start / React 18 / Vite)

  • TypeScript: Đảm bảo type safety và giảm thiểu lỗi runtime.
  • Tailwind CSS + shadcn/ui: Xây dựng giao diện responsive, hiện đại theo cơ chế Design Tokens.
  • TanStack Query (React Query): Caching và đồng bộ hóa state dữ liệu từ API server.
  • TanStack Router: Hệ thống định tuyến mạnh mẽ phía client-side.

3. Thiết Kế Cơ Sở Dữ Liệu

Cơ sở dữ liệu PostgreSQL gồm 31 bảng được quản lý phiên bản migration thông qua Flyway. Các mối quan hệ cốt lõi được cấu hình để đảm bảo tính toàn vẹn tham chiếu.

Các bảng dữ liệu quan trọng

Tên Bảng Khóa chính / Khóa ngoại Mô tả chức năng nghiệp vụ
users id Lưu trữ thông tin tài khoản người dùng, bao gồm username, password (mã hóa BCrypt), và role (ADMIN, STUDENT, TEACHER).
students id (1-1 với users.id) Chứa thông tin cá nhân của sinh viên, điểm CPA tích lũy và liên kết với ngành học (major_id).
teachers id (1-1 với users.id) Chứa thông tin giảng viên và khoa giảng dạy.
class_sections id (FK semester_id, course_id) Lớp học phần cụ thể của một môn học trong học kỳ. Lưu trữ sĩ số tối đa, sĩ số hiện tại và trạng thái khóa bảng điểm grade_locked.
class_section_schedules id (FK class_section_id, room_id, period_id) Bảng chi tiết lịch học của một lớp học phần. Hỗ trợ trường hợp lớp học nhiều buổi trong tuần (quan hệ 1-N).
enrollments id (FK student_id, class_section_id) Bản ghi đăng ký học của sinh viên. Đảm bảo ràng buộc duy nhất giữa sinh viên và lớp học phần.
grades id (FK enrollment_id) Chứa điểm chuyên cần (10%), giữa kỳ (30%), cuối kỳ (60%), và điểm thi lại retest_score. Tự động cập nhật điểm tổng kết, điểm hệ 4 và điểm chữ.
tuition_bills id (FK student_id, semester_id) Hóa đơn học phí theo kỳ của sinh viên. Chứa thông tin tổng số tiền, số tiền đã trả và trạng thái thanh toán.

4. Đăng Ký Học Phần (Kafka Queue & Concurrency)

Khi sinh viên đăng ký môn học, thay vị xử lý đồng bộ làm nghẽn luồng kết nối CSDL, hệ thống chuyển sang kiến trúc Event-Driven bất đồng bộ.

⚙️ Quy trình xử lý Đăng ký môn học
  1. Sinh viên gửi request đăng ký lớp học phần classSectionId kèm JWT.
  2. API Controller kiểm tra điều kiện cơ bản (cổng đăng ký có mở không) -> Tạo một UUID requestId -> Lưu request trạng thái PENDING vào Redis/DB.
  3. Controller đẩy message EnrollmentMessage(requestId, studentId, classSectionId) vào Kafka topic "class-registration".
  4. Backend lập tức trả về HTTP status 202 Accepted cùng requestId. Phía frontend nhận kết quả ngay mà không bị quay vòng tròn, sau đó bắt đầu cơ chế Polling (hoặc WebSocket) để cập nhật trạng thái.
  5. Kafka Consumer phía sau đọc message từ topic theo thứ tự.
  6. Consumer thực hiện Pessimistic Lock (SELECT FOR UPDATE) trên dòng lớp học phần trong DB để kiểm tra sĩ số, tránh tình trạng race condition (nhiều người cùng tranh slot cuối).
  7. Consumer kiểm tra các ràng buộc nâng cao (trùng lịch học, môn tiên quyết) -> Ghi bản ghi đăng ký enrollment vào DB -> Tăng sĩ số current_slots.
  8. Consumer cập nhật kết quả (SUCCESS/FAILED) kèm lý do chi tiết vào DB/Redis, đánh dấu requestId hoàn tất để frontend lấy thông tin hiển thị cho sinh viên.

5. Động Cơ Chat Realtime (WebSocket & Redis)

Hệ thống chat realtime được xây dựng bằng giao thức WebSocket (STOMP + SockJS), hỗ trợ 3 loại phòng: Private Chat (1-1 giữa sinh viên và giảng viên), Group Chat (nhóm tự do) và Class Group Chat (nhóm tự động tạo cho từng lớp học phần).

Bảo mật WebSocket

  • Handshake Interceptor: Xác thực JWT token được truyền qua query parameter trước khi chấp nhận nâng cấp kết nối (handshake).
  • Channel Interceptor: Chặn tất cả các frame STOMP gửi lên (SEND, SUBSCRIBE) để xác thực lại token, đồng thời kiểm tra xem người dùng đó có thuộc danh sách thành viên của phòng chat (chat_room_members) hay không, ngăn chặn việc subscribe trộm tin nhắn nhóm khác.

Khả năng Scale-out

Nếu hệ thống chạy trên nhiều instance server (cluster), kết nối WebSocket của user A ở server 1 sẽ không tự truyền được sang user B kết nối ở server 2. Dự án giải quyết bằng Redis Pub/Sub làm Message Broker trung gian. Khi server nhận tin nhắn, nó sẽ broadcast tin nhắn đó lên Redis kênh chung, các instance server khác subscribe kênh này sẽ nhận được tin nhắn và đẩy xuống cho client đang kết nối tại instance của mình.

6. Trợ Lý Học Vụ AI RAG (Retrieval-Augmented Generation)

Trợ lý AI tích hợp giúp sinh viên giải đáp thắc mắc về quy chế đào tạo, học phí, và lịch trình. Kiến trúc sử dụng mô hình RAG để tránh hiện tượng ảo giác (hallucination) của LLM.

Cơ chế Reranking phân cấp (Truth-hierarchy re-ranking)

Đặc sản công nghệ của Chatbot là thuật toán re-ranking dựa trên nguồn gốc tài liệu. Các chunks sau khi tính điểm Cosine Similarity sẽ được nhân thêm trọng số tin cậy (Priority Weights):

  • Thông báo mới nhất của trường (Priority 1): Trọng số 1.0
  • Trang của các khoa/phòng ban (Priority 2): Trọng số 0.9
  • Thông tin tuyển sinh/ngành học (Priority 3): Trọng số 0.8
  • Sổ tay sinh viên (Priority 4): Trọng số 0.7
  • Thông tin tham khảo Wikipedia (Priority 5): Trọng số 0.5

Điều này đảm bảo khi có xung đột thông tin (ví dụ: quy chế thi thay đổi ở thông báo mới so với sổ tay cũ), AI sẽ ưu tiên thông báo mới nhất để trả lời sinh viên.

7. Thanh Toán VNPay & Bảo Mật Hệ Thống

Thanh toán trực tuyến VNPay

Cơ chế tích hợp sử dụng môi trường VNPay Sandbox của ngân hàng NCB. Để chống việc sinh viên thay đổi số tiền thanh toán (thay đổi param URL), hệ thống thực hiện:

  • Mã hóa SHA512 toàn bộ tham số gửi đi cùng khóa bí mật vnp_HashSecret trên backend tạo ra chữ ký vnp_SecureHash.
  • Khi VNPay phản hồi kết quả về URL callback, backend tính toán lại chữ ký để đối soát chéo. Đồng thời, API IPN kiểm tra số tiền thực trả so với số tiền cần đóng trong hóa đơn (tuition_bills) trước khi cập nhật hóa đơn sang trạng thái COMPLETED.

Cơ chế bảo mật nâng cao

  • Stateless JWT lifecycle: Access Token thời hạn ngắn (15 phút) lưu ở bộ nhớ, Refresh Token thời hạn dài (7 ngày) lưu trong cookie HttpOnly để chống lỗi XSS, kết hợp lưu trữ session ở Redis để có thể chủ động hủy bộ token (revoke) khi logout.
  • Rate Limiting (Bucket4j): Giới hạn request theo địa chỉ IP để chống brute force tài khoản và chống tấn công từ chối dịch vụ (DDoS) vào cổng đăng ký học phần.
  • AOP Audit logging: Sử dụng Aspect Oriented Programming để đánh dấu Annotation @Audit lên các phương thức nghiệp vụ quan trọng (như sửa đổi điểm, cập nhật học phí). AOP tự động bắt request, ghi nhận thời gian, tài khoản thực hiện, nội dung thay đổi trước và sau vào bảng audit_logs bất đồng bộ, tránh làm ảnh hưởng hiệu năng xử lý chính.

8. Vòng Đời Một Học Kỳ

Hệ thống hỗ trợ quản lý trọn vẹn chu kỳ hoạt động của một học kỳ đại học qua 9 bước khép kín:

  1. Khởi tạo học kỳ: Admin mở học kỳ mới, thiết lập thời gian bắt đầu và kết thúc kỳ học.
  2. Thiết lập lớp học phần: Khai báo các lớp học phần tương ứng với môn học trong kỳ, phân công phòng học, tiết học, sĩ số tối đa và gán giảng viên phụ trách.
  3. Mở cổng đăng ký: Cho phép sinh viên đăng ký học phần trực tuyến (xếp hàng bất đồng bộ qua Kafka).
  4. Chốt cổng đăng ký học phần: Khóa các lớp học phần, chuyển trạng thái đăng ký của sinh viên từ PENDING thành REGISTERED và tự động khởi tạo bản ghi bảng điểm (grades) rỗng.
  5. Học trong kỳ & Điểm danh: Giảng viên tiến hành giảng dạy, điểm danh online (PRESENT/ABSENT/LATE) và nhập điểm thành phần chuyên cần, giữa kỳ.
  6. Đăng ký thi lại / học cải thiện: Sinh viên có điểm D/F ở học kỳ trước được quyền đăng ký thi lại hoặc học lại song song.
  7. Lập lịch thi & Chia phòng: Admin xếp ca thi, phòng thi theo môn học. Hệ thống tự động gom sinh viên lớp thường và thi lại để chia vào phòng thi tương ứng với sức chứa.
  8. Thi cuối kỳ & Nhập điểm: Tổ chức thi. Giảng viên chấm thi và nhập điểm thi cuối kỳ hoặc điểm thi lại. Hệ thống tự động tính điểm tổng kết thang 10, quy đổi điểm chữ và hệ 4 bằng JPA Listener.
  9. Khóa điểm & Đóng học kỳ: Giảng viên bấm khóa bảng điểm. Admin thực hiện chạy tác vụ tính toán kết quả học tập kỳ (GPA) và tích lũy toàn khóa (CPA) cho sinh viên, xuất báo cáo ra Excel và đóng học kỳ để lưu trữ.

🌐 Hệ Thống Hàng Đợi Kafka & Concurrency (Xử lý đồng thời)
1. Tại sao em lại chọn Apache Kafka cho chức năng đăng ký học phần mà không ghi trực tiếp xuống CSDL?
Trả lời:
- Đăng ký học phần trong đào tạo tín chỉ là bài toán chịu tải đỉnh đột biến (peak traffic load). Hàng ngàn sinh viên sẽ cùng gửi request lên trong vài giây đầu tiên mở cổng.
- Nếu ghi trực tiếp xuống CSDL PostgreSQL, mỗi request sẽ chiếm dụng một connection. Số lượng kết nối đồng thời (Connection Pool) của CSDL có giới hạn (thường là 100-200). Khi vượt quá giới hạn, DB sẽ báo lỗi timeout hoặc sập hoàn toàn, khiến giao dịch bị thất thoát và web không thể truy cập.
- Giải pháp Kafka: Kafka đóng vai trò là một bộ đệm chịu tải (Message Buffer). Khi sinh viên đăng ký, request được đưa vào Kafka dưới dạng tin nhắn. Backend trả về ngay mã trạng thái 202 Accepted. Hệ thống không bị nghẽn luồng kết nối DB. Sau đó, các Kafka Consumer sẽ lấy tin nhắn ra xử lý tuần tự phía sau, giúp bảo vệ CSDL khỏi bị quá tải và sập nguồn.
2. Nếu nhiều sinh viên cùng tranh chấp slot cuối cùng của một lớp học phần, hệ thống giải quyết thế nào để không bị vượt quá sĩ số (Over-enrollment)?
Trả lời:
Để giải quyết bài toán tranh chấp slot cuối (race condition), hệ thống áp dụng đồng thời 2 giải pháp:
1. **Kafka Queue**: Đưa các request vào hàng đợi tuần tự. Yêu cầu nào đến trước sẽ được ghi vào Kafka trước.
2. **Pessimistic Locking (Khóa bi quan)** trong database: Khi Consumer đọc tin nhắn đăng ký từ Kafka và bắt đầu ghi nhận vào CSDL, nó sẽ thực hiện truy vấn: SELECT current_slots, max_slots FROM class_sections WHERE id = :id FOR UPDATE.
Lệnh FOR UPDATE sẽ khóa dòng dữ liệu của lớp học phần đó lại. Mọi transaction khác cố gắng đọc/ghi dòng này đều phải xếp hàng chờ cho đến khi transaction hiện tại hoàn thành (commit hoặc rollback).
- Trong block transaction được khóa này, hệ thống kiểm tra: Nếu current_slots < max_slots thì mới thực hiện insert bản ghi enrollment và tăng current_slots lên 1. Nếu đã đầy, transaction rollback và trả kết quả thất bại.
Nhờ đó, sĩ số lớp luôn được kiểm soát chính xác 100% không bao giờ bị over-slots.
3. Chế độ xử lý bất đồng bộ (Asynchronous) của Kafka có thể khiến sinh viên không biết mình đã đăng ký thành công hay chưa. Em giải quyết trải nghiệm này thế nào?
Trả lời:
- Khi gửi request đăng ký thành công lên hàng đợi Kafka, backend lập tức sinh một mã UUID ngẫu nhiên gọi là requestId và lưu trạng thái đăng ký là PENDING vào Redis/Database, sau đó trả requestId về cho Frontend.
- Ở phía Frontend, hệ thống sử dụng cơ chế **Short Polling** (hoặc kết nối WebSocket): Cứ mỗi 1-2 giây, frontend sẽ gọi API GET /api/student/enrollment/status/{requestId} để hỏi trạng thái của request.
- Khi Consumer xử lý xong ở phía sau, nó sẽ cập nhật trạng thái của requestId trong Redis/DB thành SUCCESS hoặc FAILED kèm lý do cụ thể (như: Trùng lịch học, thiếu môn tiên quyết, lớp đầy).
- Frontend nhận được trạng thái cuối cùng này sẽ dừng hỏi và hiển thị thông báo đẹp mắt (Toast hoặc Modal) cho sinh viên biết kết quả chính thức.
4. Điều gì xảy ra nếu hệ thống đang chạy thì bị mất điện hoặc sập nguồn giữa chừng? Các tin nhắn đăng ký môn học trong Kafka có bị mất không?
Trả lời:
- Tin nhắn đăng ký trong Kafka **không bị mất** vì Kafka được thiết kế để ghi dữ liệu trực tiếp xuống đĩa cứng (Commit Log) thay vì lưu trên RAM như các hàng đợi in-memory khác.
- Khi tin nhắn được gửi thành công đến Kafka, nó đã được lưu trữ bền vững (Persistent).
- Hệ thống cấu hình cơ chế commit offset thủ công (Manual Acknowledge): Consumer chỉ xác nhận đã xử lý tin nhắn với Kafka (commit offset) **sau khi** đã ghi nhận thành công bản ghi đăng ký vào PostgreSQL DB.
- Nếu Consumer đang xử lý nửa chừng mà sập nguồn, offset chưa được commit. Khi hệ thống khởi động lại, Consumer sẽ đọc lại tin nhắn đó từ offset chưa commit gần nhất và xử lý lại, đảm bảo không bị thất thoát bất cứ giao dịch nào của sinh viên.
🔑 Bảo Mật, JWT Token & Redis
5. Tại sao em lại lưu trữ Refresh Token trong Cookie HttpOnly mà không lưu trong LocalStorage như Access Token?
Trả lời:
- **LocalStorage** có thể truy cập được bằng JavaScript của trình duyệt. Nếu trang web bị tấn công **XSS (Cross-Site Scripting)** (ví dụ: hacker chèn được một đoạn mã script độc hại qua ô chat hoặc comment), mã script đó có thể đọc trộm Refresh Token từ LocalStorage và gửi về server của hacker.
- **HttpOnly Cookie** là cơ chế bảo mật của trình duyệt, ngăn chặn JavaScript đọc/ghi cookie này. Trình duyệt sẽ tự động đính kèm cookie này lên các request HTTP gửi đến domain của server. Do đó, hacker không thể lấy trộm Refresh Token bằng mã lệnh JS, giúp bảo vệ an toàn cho phiên đăng nhập lâu dài của người dùng.
6. Bản chất JWT là stateless (không trạng thái), làm thế nào em thực hiện chức năng Đăng xuất (Logout) thu hồi token lập tức khi token chưa hết hạn?
Trả lời:
- Đúng là JWT nguyên bản có tính stateless, server không cần lưu trữ trạng thái token nên không thể chủ động thu hồi trước khi nó tự hết hạn (expire).
- Để giải quyết điểm yếu này, hệ thống kết hợp sử dụng **Redis**: Khi người dùng đăng nhập thành công, Refresh Token được lưu trữ vào Redis với key là username/userId.
- Khi người dùng bấm **Đăng xuất (Logout)**, backend sẽ xóa Refresh Token tương ứng khỏi Redis. Đồng thời, Access Token cũ (chỉ có thời hạn 15 phút) sẽ bị đưa vào một danh sách đen (Blacklist) trong Redis với thời gian sống bằng đúng thời gian còn lại của Access Token đó.
- Khi có request gửi lên, Filter bảo mật sẽ kiểm tra Access Token đó trong Blacklist của Redis. Nếu tìm thấy, request bị từ chối ngay lập tức với lỗi 401 Unauthorized, hoàn thành việc thu hồi token chủ động.
7. Cơ chế Rate Limiting bằng Bucket4j hoạt động ra sao và em cấu hình nó ở đâu trong ứng dụng?
Trả lời:
- **Bucket4j** hoạt động dựa trên thuật toán **Token Bucket (Thùng thẻ bài)**. Mỗi IP truy cập được cấp một chiếc thùng chứa số lượng thẻ nhất định (ví dụ: tối đa 60 thẻ). Mỗi request gửi lên sẽ tiêu tốn 1 thẻ. Thẻ mới được tự động nạp đầy lại vào thùng theo tốc độ cấu hình sẵn (ví dụ: 10 thẻ/giây). Nếu sinh viên gửi request quá nhanh làm cạn thẻ trong thùng, request sẽ bị chặn và trả về mã lỗi 429 Too Many Requests.
- Trong hệ thống, Rate Limiting được triển khai dưới dạng một **Spring Security Filter** (hoặc một Interceptor) nằm ở lớp ngoài cùng của luồng xử lý HTTP, chặn request ngay từ IP để tránh tiêu tốn tài nguyên xử lý của Controller phía sau.
💬 WebSocket Chat & Realtime Scaling
8. Làm thế nào em xác thực quyền truy cập của người dùng đối với các phòng chat WebSocket?
Trả lời:
- WebSocket không gửi kèm Header HTTP tiêu chuẩn trên từng tin nhắn sau khi đã thiết lập kết nối, vì thế việc xác thực quyền truy cập cần thực hiện qua bộ lọc chặn (Interceptors):
1. **Xác thực lúc kết nối (Connect):** Client gửi JWT token thông qua query parameter khi thực hiện bắt tay WebSocket (Handshake). Một WebSocketHandshakeInterceptor sẽ parse JWT này để xác định danh tính người dùng. Nếu token không hợp lệ hoặc hết hạn, kết nối bị từ chối ngay lập tức.
2. **Xác thực lúc đăng ký nhận tin (Subscribe):** Khi người dùng gửi lệnh SUBSCRIBE đến địa chỉ /topic/chatroom/{chatRoomId}, một WebSocketChannelInterceptor sẽ can thiệp. Interceptor này truy vấn cơ sở dữ liệu/Redis để kiểm tra xem UserId hiện tại có nằm trong bảng thành viên chat_room_members của phòng chat đó hay không. Nếu không, lệnh Subscribe bị hủy bỏ, ngăn chặn việc đọc trộm tin nhắn phòng người khác.
9. Nếu ứng dụng chat chạy trên môi trường Clustering (nhiều server backend chạy song song sau Load Balancer), làm sao tin nhắn gửi từ client kết nối ở Server 1 chuyển được đến client kết nối ở Server 2?
Trả lời:
- Nếu không cấu hình bổ sung, tin nhắn gửi lên Server 1 chỉ được phân phối cho các client đang giữ kết nối WebSocket trực tiếp đến Server 1.
- Để giải quyết bài toán scale-out này, hệ thống sử dụng **Redis Pub/Sub** làm cầu nối broker trung gian:
- Khi Server 1 nhận được tin nhắn chat từ client, thay vì chỉ gửi trực tiếp xuống WebSocket nội bộ, nó sẽ xuất bản (Publish) tin nhắn đó lên một kênh chung của Redis (ví dụ kênh: "chat-messages").
- Tất cả các instance backend (Server 1, Server 2, Server 3) đều subscribe kênh Redis này ngay khi khởi động. Khi Redis broadcast tin nhắn mới, cả Server 1 và Server 2 đều nhận được. Server 2 sẽ tìm trong danh sách WebSocket session cục bộ của mình xem có ai đang subscribe phòng chat đó không và đẩy tin xuống, đảm bảo việc truyền tin thông suốt toàn hệ thống.
🤖 Trợ Lý AI Chatbot RAG
10. Em hãy giải thích cơ chế hoạt động của hệ thống Chatbot RAG trong đồ án? Tại sao không dùng trực tiếp API của OpenAI hay Groq cho nhanh?
Trả lời:
- RAG viết tắt của **Retrieval-Augmented Generation (Thế hệ tăng cường tìm kiếm)**.
- Nếu dùng trực tiếp API của LLM (như ChatGPT, Llama), mô hình chỉ trả lời được bằng tri thức chung có sẵn trong quá trình huấn luyện của nó. Nó hoàn toàn không biết thông tin nội bộ của Đại học Thăng Long (như thời hạn nộp học phí học kỳ này, quy chế tính điểm F của trường, các ngành đào tạo cụ thể). Nếu hỏi, LLM sẽ tự bịa ra câu trả lời sai lệch (ảo giác AI).
- **Cơ chế hoạt động của RAG trong đồ án:**
1. **Tri thức hóa văn bản:** Các file quy chế, thông báo được Admin cắt thành các đoạn nhỏ (Knowledge Chunks) và chuyển đổi thành các chuỗi số (Vector Embeddings) thông qua mô hình Embedding, sau đó lưu vào CSDL PostgreSQL.
2. **Tìm kiếm ngữ cảnh:** Khi sinh viên hỏi câu hỏi (ví dụ: "Bị điểm F môn học thì làm sao?"), hệ thống chuyển câu hỏi đó thành vector và tính toán độ tương đồng Cosine Similarity in-memory với các vector tri thức trong DB để lọc ra 5 đoạn văn bản liên quan nhất.
3. **Tạo prompt ngữ cảnh:** Hệ thống ghép 5 đoạn văn bản đó làm ngữ cảnh (Context) cùng với câu hỏi ban đầu tạo thành một System Prompt gửi đến Groq API (sử dụng mô hình Llama 3.3).
4. **LLM tổng hợp câu trả lời:** Llama 3.3 đọc ngữ cảnh được cung cấp và viết câu trả lời chính xác, đúng sự thật theo quy định của nhà trường.
11. Cơ chế reranking phân cấp độ tin cậy (Truth-hierarchy re-ranking) giải quyết vấn đề gì trong chatbot?
Trả lời:
- Trong môi trường đại học, các quy định có thể thay đổi theo thời gian. Ví dụ, trong Sổ tay sinh viên xuất bản năm 2024 quy định học phí thi lại là 100k/môn, nhưng Thông báo khẩn của nhà trường năm 2026 tăng lên thành 150k/môn.
- Nếu chỉ tính độ tương đồng cosine thông thường, hai đoạn văn bản này đều có điểm tương đồng rất cao với câu hỏi "Học phí thi lại là bao nhiêu?". AI sẽ bối rối hoặc trả lời cả hai con số.
- **Giải pháp:** Hệ thống đưa vào trọng số ưu tiên Priority Weights dựa trên nguồn gốc tài liệu. Các chunks trích xuất từ **Thông báo mới (Priority 1)** được nhân trọng số 1.0 (giữ nguyên điểm similarity), trong khi chunks trích xuất từ **Sổ tay sinh viên (Priority 4)** chỉ nhân trọng số 0.7 (giảm điểm similarity xuống).
Nhờ đó, thông tin chính thống và mới nhất luôn được đưa lên đầu ngữ cảnh cung cấp cho LLM, giúp chatbot đưa ra câu trả lời chuẩn xác nhất theo dòng thời gian.

Giải Thích Thuật Ngữ Chuyên Ngành

Giúp nắm rõ các định nghĩa, vai trò thực tế và cách trả lời ghi điểm tuyệt đối trước Hội đồng bảo vệ đồ án.

⚡ TanStack Query (React Query)

Định nghĩa: Thư viện quản lý các trạng thái bất đồng bộ (Async State) như fetching, caching, đồng bộ và làm mới dữ liệu giữa Client (React) và API Server.

Vai trò trong đồ án: Quản lý toàn bộ dữ liệu gọi từ API (thời khóa biểu, hóa đơn học phí, danh sách điểm). React Query lưu trữ dữ liệu vào bộ nhớ đệm (cache). Khi người dùng chuyển trang rồi quay lại, dữ liệu được lấy ngay từ cache ra hiển thị lập tức mà không phải gọi lại API, tăng tốc độ trải nghiệm web rõ rệt.

Keyword ăn điểm: Server State, Caching, Stale-While-Revalidate, Background Refetching.

🗺️ TanStack Router

Định nghĩa: Thư viện quản lý định tuyến chuyển trang thế hệ mới cho React, hỗ trợ định tuyến dựa trên cấu trúc file (File-based Routing) và an toàn kiểu dữ liệu (Fully Type-Safe Routing).

Vai trò trong đồ án: Quản lý bản đồ các trang của ứng dụng (Student Dashboard, Admin Panel, Registration...). Tính năng Type-Safe giúp compiler báo lỗi ngay lập tức khi code nếu viết sai URL hoặc truyền thiếu tham số, triệt tiêu 100% lỗi broken links (liên kết hỏng) lúc chạy thực tế.

Keyword ăn điểm: Type-Safe Routing, File-based Route, Prefetching, Client-side Navigation.

🧠 AI RAG (Retrieval-Augmented Generation)

Định nghĩa: Kỹ thuật kết hợp mô hình ngôn ngữ lớn (LLM Llama 3.3) với một hệ thống truy xuất dữ liệu độc lập. Hệ thống đi tìm ngữ cảnh liên quan trong CSDL trước, sau đó kẹp dữ liệu này cùng câu hỏi để LLM viết câu trả lời.

Vai trò trong đồ án: Trợ lý AI hỗ trợ tư vấn quy chế. Thay vì để Llama tự đoán mò quy định của Thăng Long (dễ bị ảo giác AI), RAG tìm kiếm chính xác điều khoản trong CSDL văn bản của trường và hướng dẫn AI viết câu trả lời hoàn toàn chính xác theo quy định thực tế.

Keyword ăn điểm: Hallucination Control, Vector Embedding, Cosine Similarity, Knowledge Chunks.

✂️ Aspect Oriented Programming (AOP)

Định nghĩa: Lập trình hướng khía cạnh. Phương pháp lập trình tách các nghiệp vụ bổ trợ lặp đi lặp lại ở nhiều nơi (như kiểm tra bảo mật, ghi nhật ký log) ra khỏi mã nguồn nghiệp vụ chính.

Vai trò trong đồ án: Dùng để ghi vết Audit Logs (nhật ký sửa điểm, khóa điểm). Thay vì viết code kết nối DB để lưu log ở mọi Controller/Service nghiệp vụ, AOP tách phần code lưu log ra một khía cạnh riêng (Aspect) và chèn tự động qua Annotation @Audit, giúp code nghiệp vụ cực kỳ sạch sẽ.

Keyword ăn điểm: Cross-cutting concerns, Aspect, Advice, Clean Code (Separation of Concerns).

📁 Phiên bản hóa CSDL (Flyway Migration)

Định nghĩa: Công cụ theo dõi, quản lý và áp dụng lịch sử thay đổi cấu trúc bảng (database schema) theo thời gian thông qua các file SQL đánh số phiên bản thứ tự (V1, V2, V3...).

Vai trò trong đồ án: Khi dự án thay đổi cấu trúc bảng (ví dụ thêm lịch schedules, thêm cột điểm thi lại), ta viết script SQL. Flyway tự động phát hiện và chạy script này khi backend Spring Boot khởi động, giữ cho database của tất cả máy dev và server deploy luôn đồng bộ 100%.

Keyword ăn điểm: Database Version Control, Schema Migration, Flyway Schema History.

🔒 Pessimistic Lock (Khóa bi quan)

Định nghĩa: Cơ chế khóa độc quyền dòng dữ liệu trong database khi một transaction truy cập vào, chặn tất cả các request song song khác không được đọc/ghi dữ liệu đó cho đến khi transaction đầu tiên commit.

Vai trò trong đồ án: Áp dụng tại luồng đăng ký học phần (Kafka Consumer). Khi Consumer bắt đầu kiểm tra sĩ số và ghi nhận sinh viên vào lớp, nó dùng SELECT FOR UPDATE để khóa dòng lớp học phần đó lại. Tránh tình trạng race condition (nhiều request cùng ghi nhận slot trống cuối cùng dẫn đến vượt sĩ số tối đa).

Keyword ăn điểm: SELECT FOR UPDATE, Race Condition Prevention, Data Integrity, Transaction Isolation.
